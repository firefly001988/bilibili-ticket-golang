# 使用手册

本手册不讲原理，只讲操作。每一步告诉你点哪里、填什么。

---

## 📋 快速上手（三分钟入门）

如果你是老手，只想快速跑通流程：

```
① 启动程序 → 账号管理页扫码登录大号
② 确认 Worker 管理页 本机 Worker 🟢 绿色
③ 左侧任务组 → 创建新组（如"CP30"）
④ 点进任务组 → 输入项目 ID 查询 → 添加 SKU
⑤ 配置任务组：选账号、选 Worker
⑥ 添加购票组 → 点「等待开始准点抢票」
```

> 以下各章节有每一步的详细操作。

---

## 一、启动与登录

### 1.1 首次启动

1. 双击主程序图标，等待界面加载完毕。
2. 首次启动会弹出**语言选择**窗口，选「简体中文」。
3. 点击账号管理页的 **扫码登录**，用 Bilibili App 扫码。

### 1.2 确认本机 Worker 已连接

看一眼左下角状态栏：

| 指示灯 | 含义 | 下一步 |
|--------|------|--------|
| 🟢 绿色（已连接） | Worker 正常运行 | ✅ 继续往下看 |
| 🔴 红色 / ⚫ 灰色 | Worker 没起来 | 看下面排查 |

**Worker 自启动规则：**

| 情况 | 会不会自启 |
|------|-----------|
| 第一次用，数据库没有 `local` 记录 | ✅ 自动创建并拉起 |
| 上次正常退出，这次旧进程已死 | ✅ 自动恢复并重新监听端口 |
| 旧进程还在跑（端口被占用） | ❌ 启动失败，需手动结束占用进程后重启程序 |

**排查二步（Worker 没起来时依次检查）：**

1. 端口 `127.0.0.1:37900` 是否被占用？（`netstat -ano | findstr :37900`）
2. `data/local/` 目录是否存在、有写入权限？

## 二、账号管理

### 2.1 添加新账号

#### 方式 A：扫码添加（推荐）

1. 左侧菜单 **账号管理** → 点 **添加账号** → **扫码登录**。
2. 填写**备注名**（如"小号1"）。
3. 点 **生成二维码**，打开 Bilibili App 扫码。
4. 登录成功后，该账号下的购票人会自动同步进来。

#### 方式 B：导入凭据

1. **账号管理** → **添加账号** → **导入账号**。
2. 粘贴 JSON 凭据文档（必须含 `SESSDATA` 和 `bili_jct`）。
3. 点确认。

### 2.2 账号标签

1. 在账号列表找到目标账号，点 **编辑标签**。
2. 输入标签（如 `主力`、`华东`），回车确认，支持逗号或换行分隔。
3. 保存。标签可用于任务组账号筛选。

## 三、购票人管理

购票人 = 你要帮谁抢票（姓名 + 身份证号 + 手机号）。数据来自 Bilibili 账号的实名购票人。

### 3.1 同步购票人

| 操作 | 入口 | 说明 |
|------|------|------|
| **同步单个账号** | 账号管理 → 找到账号 → 点「同步购票人」 | 拉取该账号下所有购票人 |
| **快速同步全部** | 左侧菜单 **购票人** → 点「快速同步购票人」 | 所有账号同时拉取，同名同证件自动合并 |
| **完整同步全部** | 左侧菜单 **购票人** → 点「从全部账号刷新购票人」 | 同上，但逐个账号顺序拉取 |

> 信息带 `*` 号（打码）说明 Bilibili 未返回完整数据，稍后再试。

### 3.2 把购票人复制到其他账号

购票人"张三"只在账号 A 里有，想让账号 B 也能用他：

1. **购票人列表** → 找到"张三" → 点 **同步到账号**。
2. 选择目标账号，系统自动在目标账号的 Bilibili 上创建购票人。
3. 也可以点 **同步到全部账号**，一键复制到所有已启用账号。

### 3.3 修改手机号

1. 购票人列表 → 找到目标 → 点 **统一修改手机号**。
2. 输入新手机号保存。此操作仅更新本地存储，不会同步到 Bilibili。

### 3.4 删除购票人

1. 购票人列表 → 点 **删除**。
2. 可选：指定从哪些账号删除（留空=从全部账号删除）。
3. 确认删除。

## 四、Worker 管理

### 4.1 本机 Worker（自动管理）

本机自带 Worker（叫 `local`），程序启动时自动拉起，**无需手动操作**。

> `local` Worker 不能手动删除、断开或编辑。

### 4.2 添加远程 Worker

#### 步骤 ①：在雇主端生成配置

1. 左侧菜单 **Worker 管理** → 点 **生成配置**。
2. 填写：

   | 字段 | 说明 | 示例 |
   |------|------|------|
   | **Worker ID** | 唯一标识 | `home-server` |
   | **监听地址** | Worker 监听地址 | `0.0.0.0:18080` |
   | **TLS 主机名** | 远程机器 IP 或域名，逗号分隔 | `192.168.1.100,myworker.example.com` |

3. 点**生成**，得到一串 Base4096 编码的长字符串。复制它。
4. 可选：点 **导入该 Worker** 可立刻填入导入窗口。

#### 步骤 ②：在远程机器上启动

```bash
# 导入配置（生成 worker.json）
ticket-worker import "那串长字符串"

# 或用管道
echo "那串长字符串" | ticket-worker import --stdin

# 指定输出目录（默认 data/worker）
ticket-worker import -o /path/to/output "那串长字符串"

# 启动 Worker
ticket-worker serve --config data/worker/worker.json
```

> 其他子命令：`ticket-worker run --task <task.json>` 执行单次抢票任务（调试用）；`ticket-worker version` 查看版本信息。

看到日志 `listening on ...` 即启动成功。

#### 步骤 ③：在雇主端连接

1. **Worker 管理** → 点 **导入 Worker**。
2. 粘贴长字符串，点导入。
3. 系统自动配置 TLS 并连接，几秒内变 🟢 绿色。
4. 如果连接失败：
   - 检查远程防火墙是否放行端口
   - `ping` 确认网络互通
   - 确认远程 Worker 进程还在跑

### 4.3 批量部署远程 Worker（SSH）

如果你有多台机器，可以通过 SSH 自动部署：

1. **Worker 管理** → 点 **批量部署**。
2. 添加目标服务器（IP、SSH 端口、用户名、密码/私钥）。
3. 选择 Worker 二进制来源（本地上传 或 远程下载）。
4. 配置安装目录、Worker 端口、启动方式（nohup / systemd-user）等高级选项。
5. 点 **开始部署**，进度实时显示。

> SSH 密码仅在本次部署中使用，不会保存。

### 4.4 日常管理

| 操作 | 什么时候用 | 效果 |
|------|-----------|------|
| **断开** | 暂时不用，保留配置 | 不会自动重连 |
| **重连** | 断开后重新连回来 | 最多重试 5 次 |
| **编辑** | 改地址、改标签 | 有任务在跑时不能编辑 |
| **删除** | 彻底移除 | 有任务在跑时不能删除 |
| **编辑标签** | 给 Worker 打标签 | 标签可用于任务组 Worker 筛选 |

> 协议版本不一致时，Worker 会显示「版本被拒绝」。可点 **强制连接** 跳过版本校验（不安全）。

### 4.5 时钟校准

在 Worker 管理页选中一个可达的 Worker，详情区会显示该 Worker 测得的本地时间与 Bilibili API / NTP 服务器的时钟偏移。页面每 5 秒自动刷新展示，Worker 侧偏移测量结果缓存约 2 分钟后重新校准。抢票瞄准时间会自动按此偏移修正。

> 偏移绝对值超过 1000ms 会标红，建议排查本机或远程机器的时钟同步。

## 五、创建抢票任务

核心流程：

```
创建任务组 → 查项目加 SKU → 选账号和 Worker → 添加购票组 → 启动
```

### 5.1 创建任务组

1. 左侧菜单最下方 **任务组** 区域 → 在输入框输入名称（如"CP30"）→ 回车或点 `+`。
2. 任务组出现在列表中，点它进入详情页。

### 5.2 添加 SKU（宏任务）

一个 SKU = 一个具体票档。一个任务组可以添加多个 SKU。

1. 在任务组详情页，点 **添加 SKU（查项目）**。
2. 在弹出窗口中输入**项目 ID**（Bilibili 商品页 URL 里的数字），点查询。
3. 系统展示该项目下所有场次和票种。找到你要的票种，点 **选择**。
4. 确认活动日期和票种信息，点 **确认添加**。
5. SKU 出现在列表中。可以重复此步骤添加多个 SKU。

> 每个 SKU 支持单独设置：自定义开抢时间、主力 Worker、后备 Worker。

### 5.3 配置任务组（选账号和 Worker）

在任务组详情页的上方，点 **任务组调度配置**：

| 字段 | 说明 | 默认 |
|------|------|------|
| **账号池** | 任务组独占这些账号，调度只从这里选 | 必填 |
| **主力 Worker** | 优先分配任务 | 必填 |
| **后备 Worker** | 主力故障时才顶上 | 可选 |
| **订单支付时限** | 每波抢票后等待支付的时间 | 10 分钟 |
| **抢票波次时长** | 每波持续抢票时间 | 3 分钟 |
| **最大抢票波次** | 总共跑几波 | 3 次 |

> 配置后点保存。任务组运行中配置锁定，不能修改。

### 5.4 添加购票组

一个购票组 = 一组要坐在一起的人（如"张三+李四"两张连座）。

1. 在 SKU 列表中展开一个 SKU，找到 **购票组** 区域。
2. 从下方购票人列表勾选你要的人（不超过订单人数上限）。
3. （可选）勾选 **回流拆分** — 回流阶段允许拆成单人下单。
4. （可选）设置 **权重** 和 **优先级**。
5. 点 **添加购票组**。

一个 SKU 可以加多个购票组。准点阶段系统会尽量把多个组合并到一笔订单。

### 5.5 启动抢票

**启动前确认：**
- ✅ 任务组已配置账号池和 Worker
- ✅ 至少有一个 SKU
- ✅ 每个 SKU 有至少一个购票组
- ✅ 所有 SKU 的活动日期已确认（`needsReview` 为 false）

点 **等待开始准点抢票**。

系统会自动完成：
```
Planner 生成 Intent → Dispatcher 分配 Worker 和账号 → Worker 开始循环下单
```

> 任务组启动后，点右侧的「调度日志」和「集群日志」查看实时状态。

### 5.6 回流模式

准点没抢到？在所有准点 Attempt 停止后：

1. 点 **立刻开始回流抢票**。
2. 系统自动用回流模式重新规划（允许拆单、单人匹配）。
3. 自动分配新的 Attempt，开始刷漏票。

### 5.7 停止与强制操作

| 操作 | 按钮 | 效果 |
|------|------|------|
| **停止** | 停止本任务组 | 停止所有活跃 Attempt，释放 Worker |
| **强制停止并重置** | 强制停止并重置 | 即使有成功订单也强制重置 |
| **强制重启** | 强制重启 | 停止 → 重置 → 立刻重新启动 |

## 六、监控与干预

### 6.1 调度日志

左侧菜单 **集群** → **调度日志**。

| 区域 | 内容 |
|------|------|
| 左侧列表 | 所有 Attempt（执行记录），支持按状态筛选 |
| 右侧日志 | 选中一个 Attempt 查看详细执行日志 |
| 顶部自动刷新 | 默认 5 秒自动刷新，可手动切换 |

**删除记录：** 选中已结束的 Attempt（成功/失败/已取消），点「删除选中」。

### 6.2 集群日志

左侧菜单 **集群** → **集群日志**。

记录所有集群级别的事件：Worker 连接/断开、健康状态变化、心跳超时、任务完成/失败、调度信息等。

支持：
- **搜索** — 按 Worker、阶段、事件类型、消息搜索
- **清空** — 一键清空所有集群日志，便于重新开始跟踪

### 6.3 订单记录

左侧菜单 **集群** → **订单记录**。

所有下单成功的订单，显示项目/场次/SKU、购票人、账号/Worker、支付截止时间。

支持：
- **付款** — 打开支付二维码窗口
- **复制支付链接** — 手动转发给购票人
- **搜索** — 按订单号、项目名、购票人等搜索

### 6.4 Attempt 状态说明

| 状态 | 含义 |
|------|------|
| `queued` | 已排队，等待分配 Worker |
| `waiting` | 已下发到 Worker，等待到达开抢时间 |
| `running` | Worker 正在循环调用 Bilibili 下单接口 |
| `stopping` | 正在取消中 |
| `stopped` | 已被手动停止 |
| `succeeded` | 下单成功，已有订单号 |
| `failed` | 失败（原因见 reason 字段） |

### 6.5 Intent 状态说明

调度日志中每个 Intent 展示以下字段：

| 字段 | 含义 |
|------|------|
| `armed` | ✅ 已武装，可以分配 Attempt |
| `succeeded` | 该 Intent 已有成功的订单 |
| `terminal` | 已终止（成功或失败后不再分配） |
| `activeCount` | 当前正在运行的 Attempt 数量 |
| `deficit` | 还需分配的 Attempt 数量 |
| `failureReason` | 失败原因 |

### 6.6 通知

下单成功时弹出系统通知。如果配置了 Bark / Gotify / PushPlus 等渠道，也会推送到对应渠道。

**管理通知渠道：** 左侧菜单 **通知**。

| 操作 | 说明 |
|------|------|
| **添加渠道** | 选择渠道类型（Bark / Gotify / PushPlus 等），填写名称和对应参数，保存 |
| **编辑** | 修改已有渠道的参数 |
| **测试** | 发送一条测试消息，验证渠道是否配置正确 |
| **删除** | 移除不再使用的渠道 |

### 6.7 支付二维码

下单成功后自动弹出支付二维码窗口。如果关闭了，可以：
- **订单记录** → 点 **付款**
- 支付链接也可以复制后自行打开

## 七、设置

左侧菜单最下方 **设置**。

### 7.1 语言

切换界面语言（简体中文 / English）。选择后立即生效，并记住选择，下次启动自动应用。

### 7.2 重试间隔

每次抢票尝试之间的等待时间（毫秒），可调范围 50–2000ms，默认 500ms。

| 区间 | 说明 |
|------|------|
| < 500ms | 低延迟模式，高频率可能触发风控（黄色警告） |
| ≥ 500ms | 保守模式，风控风险较小（信息提示） |
| ≥ 1000ms | 高间隔警告，频率过低可能错过抢票时机（红色警告） |

> 保存后立即生效，包括正在运行的抢票任务。

### 7.3 启动延时

进入抢票循环前的固定等待时间（毫秒），可调范围 0–500ms，默认 0ms。设为 0 立即开始。

可用于错开多个任务的启动时机，避免瞬时请求风暴。

| 区间 | 说明 |
|------|------|
| 0ms | 立即开始 |
| ≤ 100ms | 轻微延时 |
| > 100ms | 较长延时，错峰启动 |

### 7.4 购票人管理 Worker

选择哪些 Worker 用于读取账号购票人和同步购票人。默认使用本机 `local` Worker。选择多个 Worker 后，多账号同步可以并行执行。

### 7.5 关于

显示程序版本、重试间隔与启动延时的默认值和取值范围等说明信息。

## 八、常见问题

### 问题速查

| 问题 | 检查 |
|------|------|
| 本机 Worker 红色 | ① 排查二步（见第一章） ② 手动重启程序 |
| 远程 Worker 连不上 | ① 防火墙放行端口 ② ping 通吗 ③ Worker 进程活着吗 |
| 提示「版本不匹配」 | Worker 与雇主版本不一致，点强制连接（不安全） |
| Attempt 一直 queued | ① 有健康 Worker 吗 ② 有可用账号吗 ③ 账号冷却中吗 |
| Attempt failed | 点开看 reason 字段 |
| 购票人带 `***` | Bilibili 没返回完整信息，稍后重新同步 |
| 验证码触发 | 系统自动换 Worker 重试，无需干预 |

### 系统自动处理

| 故障 | 自动处理 |
|------|---------|
| Cookie 过期 | 换另一个可用账号 |
| Worker 挂了 | 把 Attempt 挪到其他健康 Worker |
| 触发验证码 | 换 Worker 重试 |
| 账号被风控 | 冷却 5 分钟后再用 |
| 心跳超时 15 秒 | 判定 Worker 失联，触发故障转移 |

**你只需要确保：** 足够多的主力账号 + 健康可用的 Worker。

## 九、概念速查

| 概念 | 一句话解释 |
|------|-----------|
| **雇主** | 你的桌面程序，管账号、管 Worker、做规划、发任务 |
| **Worker** | 实际跑抢票的程序，本机或远程都行 |
| **任务组** | 一组抢票任务的集合，独占账号和 Worker |
| **宏任务（SKU）** | 一个具体票档的抢票配置 |
| **购票组** | 一组要坐在一起的人 |
| **Intent** | Planner 生成的"用某某人去抢某个 SKU"的意图 |
| **Attempt** | 一个 Intent 绑一个账号 + 一个 Worker，实际去跑 |
| **Phase** | 阶段：`punctual`（准点）或 `reflow`（回流） |
| **准点** | 卡着开售时间下单，购票组尽量合并 |
| **回流** | 准点没抢到，拆成单人继续刷漏票 |
| **波次** | 回流阶段的每一轮抢票，默认 3 轮，每轮 3 分钟 |
| **标签** | 账号/Worker 的用户自定义标签，用于任务组筛选 |
| **冷却** | 账号或 Worker 因风控/故障暂时禁用，自动恢复 |
