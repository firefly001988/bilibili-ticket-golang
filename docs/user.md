# 使用手册

本手册不讲原理，只讲操作。每一步告诉你点哪里、填什么。

---

## 一、准备工作

1. 启动程序后，先在主界面用你的 Bilibili 大号扫码登录。这是雇主端的主账号。
2. 确认左下角状态栏的「本机 Worker」显示绿色（已连接）。这是自动启动的，你不需要管。

### 本机 Worker 什么情况下会自启动？

程序启动时，只要本地 Worker 不存在或已挂掉，就会自动拉起一个新的：

| 情况 | 会自启吗 |
| --- | --- |
| 第一次用，数据库里没有 `local` 这条记录 | ✅ 是 |
| 上次正常退出，这次启动时旧进程已死 | ✅ 是 |
| 旧进程还在跑（比如上次异常退出没杀干净） | ❌ 复用已有的，不重启 |

**如果本机 Worker 没起来（显示红色/灰色），检查这三样：**

1. `ticket-worker(.exe)` 是否和主程序在同一目录下。
2. 端口 `127.0.0.1:18080` 是否被其他程序占用。
3. `data/local-worker/` 目录是否有写入权限。

## 二、添加账号

一个号不够用？可以再加。每个账号独立拥有自己的 Cookie 和设备指纹，互不干扰。

### 方式 A：扫码添加

1. 左侧菜单点 **账号管理**。
2. 点 **添加账号** → **扫码登录**。
3. 填写账号备注名（随便起，比如"小号1"）。
4. 选择角色：
   - **主力**：正常参与抢票。
   - **后备**：平时不干活，主力账号出问题时才顶上。
5. 点 **生成二维码**，用 Bilibili 客户端扫码确认。
6. 登录成功后，该账号下的购票人会自动同步进来。

### 方式 B：导入凭据 JSON

如果你有导出的凭据文件（其他工具或浏览器插件导出）：

1. **账号管理** → **添加账号** → **导入凭据**。
2. 粘贴 JSON 内容（必须包含 `cookies`，至少要有 `SESSDATA` 和 `bili_jct`）。
3. 选择角色，确认。

### 补充：同步主账号

如果你在主界面重新登录了大号，点一下 **同步主账号**，雇主会把最新凭据更新到账号池。

## 三、同步购票人

购票人就是你要帮谁抢票（姓名 + 身份证号 + 手机号）。这些数据来自你 Bilibili 账号里已添加的实名购票人。

### 同步单个账号的购票人

1. **账号管理** → 找到目标账号 → 点 **同步购票人**。
2. 系统会拉取该账号下的所有实名购票人，保存完整信息（身份证号、手机号）。
3. 如果某个购票人的信息还是打码的（带 `*` 号），说明 Bilibili 还没返回完整数据，稍后再试。

### 一键同步所有账号的购票人

**账号管理** → 点 **同步全部购票人**。所有启用账号的购票人都会拉下来，同名同证件的自动合并为一条。

### 把一个购票人复制到其他账号

假设购票人"张三"只在账号 A 里有，你想让账号 B 也能用他抢票：

1. **购票人列表** → 找到"张三" → 点 **同步到账号**。
2. 选择目标账号 B。
3. 系统会自动在账号 B 的 Bilibili 上创建购票人"张三"（姓名、电话、身份证号、证件类型完全一致）。

也可以直接点 **同步到全部账号**，一键复制到所有已启用的账号。

> 注意：这个操作实际上是在 Bilibili 上创建购票人，会调用 Bilibili 的接口。每个账号的购票人数量有上限。

## 四、添加远程 Worker

本机已经自带一个 Worker（叫 `local`），不需要你管。如果你还有另一台电脑想一起抢票，那台电脑就是**远程 Worker**。

### 步骤 1：在雇主端生成配置

1. 左侧菜单点 **Worker 管理**。
2. 点 **生成远程配置**。
3. 填写：
   - **Worker ID**：给这台机器起个名，比如 `home-server`。
   - **监听地址**：Worker 在远程机器上监听的地址，默认 `0.0.0.0:18080` 就行。
   - **主机名/IP**：远程机器的 IP 或域名，多个用逗号分隔。比如 `192.168.1.100,myworker.example.com`。
4. 点生成，你会得到一串**长字符串**（Base4096 编码）。复制它。

### 步骤 2：在远程机器上启动 Worker

1. 把 `ticket-worker` 程序复制到远程机器。
2. 在那台机器上新建一个文本文件 `worker.json`，把刚才复制的长字符串**解码**后写入。或者直接在远程机器的终端里运行程序，让它自己解码。
3. 运行：
   ```
   ticket-worker serve --config worker.json
   ```
4. 看到日志输出 `listening on ...` 就说明启动成功了。

### 步骤 3：在雇主端添加 Worker

1. 回到雇主端的 **Worker 管理**。
2. 点 **导入远程配置**，粘贴刚才那串长字符串。
3. 系统会自动配置 TLS 证书并尝试连接。如果远程 Worker 已经在运行，几秒内就会显示绿色。
4. 如果连接失败，检查远程机器的防火墙是否放行了对应端口，网络是否互通。

### 日常管理

| 操作 | 干什么用 |
| --- | --- |
| **断开** | 暂时不用这个 Worker，保留配置，不会自动重连。 |
| **重连** | 断开后想重新连回来，点这个。最多重试 5 次。 |
| **编辑** | 改地址、改角色。注意：Worker 正在跑任务时不能编辑。 |
| **删除** | 彻底删掉这个 Worker。同样，有任务在跑时不能删。 |

> `local` Worker 是自动管理的，不能手动删、断、改。

## 五、创建抢票任务

这是核心流程：查场次 → 建宏任务 → 加购票组 → 确认活动日期 → 启动。

### 5.1 查场次

1. 左侧菜单点 **场次查询**。
2. 输入项目 ID（Bilibili 商品页 URL 里的数字），点查询。
3. 列表会展示该项目下的所有票档。找到你要抢的那一档，记下它的信息。

### 5.2 新建宏任务

一个宏任务 = 一个 SKU（具体票档）的抢票配置。

1. 点 **任务管理** → **新建宏任务**。
2. 填写：
   - **任务组**：先建一个任务组（随便起名，比如"CP30"），然后把宏任务归到这个组下。一个任务组可以放多个宏任务。
   - **项目 ID / 场次 ID / SKU ID**：从场次查询那里复制过来。
   - **单订单人数上限**：一般不用改，从 API 自动获取。如果 API 没返回，默认 4。
   - **副本数**：一个 Intent 用不同账号同时发几个 Attempt。默认 1 就行。
   - **智能合并**：开启后，多个购票组会尽量拼满一辆车（best-fit-decreasing）。一般开着。
   - **活动日期**：必须手动选择并确认（点「确认日期」），不确认不能启动。
3. 保存。

> 旧版迁移过来的宏任务会标记「待审核」（needsReview）。你需要逐个确认：检查活动日期是否正确、购票组是否合理，然后点「确认日期」。

### 5.3 添加购票组

一个购票组 = 一组要一起买票的人（比如"张三+李四"两张连座）。

1. 在宏任务详情里点 **添加购票组**。
2. 从购票人列表中勾选要添加的人。人数不能超过单订单上限。
3. 可选：勾选 **允许拆分**（回流阶段允许把这组人拆成单人下单）。
4. 保存。

一个宏任务可以有多个购票组。准点阶段系统会尽量把多个组合并到一辆车里。

### 5.4 启动抢票

1. 确认宏任务的 **活动日期已确认**、**有待审核** 已关闭、有至少一个购票组。
2. 点 **开始抢票（准点）**。
3. 系统会：
   - 调用 Planner 生成订单 Intent。
   - Dispatcher 挑选健康的 Worker 和可用的账号。
   - 把 Attempt 下发到 Worker 开始循环下单。
4. 在 **Attempt 列表** 里可以实时看到每个 Attempt 的状态。

### 5.5 切换到回流

准点没抢到？等所有准点 Attempt 都停了之后，点 **切换到回流**。系统会自动用回流模式重新规划（允许拆单、单人匹配），然后重新分配 Attempt。

## 六、监控与干预

### 查看任务状态

**任务管理** → 点开宏任务，下方有三个 Tab：

| Tab | 内容 |
| --- | --- |
| 购票组 | 当前宏任务挂了多少个购票组 |
| Intent | Planner 生成的订单意图，以及每个 Intent 的武装状态 |
| Attempt | 实际下发到 Worker 的执行尝试，显示状态、账号、Worker、订单号 |

Attempt 状态说明：

| 状态 | 含义 |
| --- | --- |
| `queued` | 已排队，等待分配 Worker |
| `waiting` | 已下发到 Worker，等待到达 startAt 时间 |
| `running` | Worker 正在循环调用 Bilibili 下单接口 |
| `stopping` | 正在取消 |
| `stopped` | 已被手动停止 |
| `succeeded` | 下单成功，已有订单号 |
| `failed` | 失败（原因见 reason 字段） |

### 查看详细日志

在 Attempt 上点 **查看日志**，可以看到 Worker 返回的每一步日志：下单请求、返回码、验证码触发、重试等。

### 手动停止

- **停止单个 Attempt**：在 Attempt 列表里点停止。
- **停止整个宏任务**：在宏任务上点 **停止**，会停止该宏任务的所有活跃 Attempt。

### 通知

下单成功时，雇主会弹出通知（如果之前配置了 Bark/Gotify/PushPlus 等通知渠道，也会推送到对应渠道）。

## 七、故障会自己处理

你不用操心下面这些事，系统会自动处理：

- **Cookie 过期** → 换另一个同角色的账号。
- **Worker 挂了** → 自动把 Attempt 挪到另一个健康的 Worker。
- **触发验证码** → 换 Worker 重试。
- **账号被风控** → 冷却 5 分钟后再考虑使用。
- **心跳断了** → 15 秒超时，判定 Worker 失联，触发故障转移。

你只需要确保：有足够多的主力账号 + 有健康可用的 Worker。

## 八、概念速查

| 概念 | 一句话解释 |
| --- | --- |
| **雇主** | 你的桌面程序，负责管账号、管 Worker、做规划、发任务。 |
| **Worker** | 实际执行抢票的程序，可以在本机也可以在其他电脑上。 |
| **宏任务** | 一个 SKU（票档）的抢票计划，包含活动日期、人数上限、副本数等配置。 |
| **购票组** | 一组要一起买的人。多个购票组在准点阶段会尽量拼车。 |
| **Intent** | Planner 生成的"我要用某几个购票人抢某个 SKU 的票"的意图。 |
| **Attempt** | Intent 的一次执行尝试，绑定到某个账号 + 某台 Worker。一个 Intent 可以有多个 Attempt（副本）。 |
| `desiredReplicas` | 每个 Intent 要创建几个 Attempt（用不同账号/Worker 并行抢）。 |
| `hardConcurrency` | 同一个宏任务的 Intent 最多同时跑几个 Attempt。 |
| **准点** | 按 Bilibili 的开售时间精确卡点，购票组尽量合并。 |
| **回流** | 准点没抢到之后，拆成单人继续刷漏票。 |
