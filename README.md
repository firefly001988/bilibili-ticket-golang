# Bilibili-ticket-golang

一个采用 **Employer–Worker 架构**的 Bilibili 会员购辅助工具. 桌面端作为雇主控制面, 负责账号, 购票人, 任务规划, Worker 调度, 订单记录和支付提示; 本机或远程 Worker 负责执行会员购循环下单.

雇主端基于 [Wails v3](https://wails.io/) + Go + Vue/Vuetify 构建, 任务和订单数据持久化到 SQLite. 雇主与 Worker 之间使用 **gRPC + mTLS** 通信, 支持本机 Worker 自动管理和远程 Worker 扩展; BWS 任务仍由桌面端执行.

集群部署, 通信协议, 调度语义和安全设计详见 [docs/cluster.md](docs/cluster.md), 使用流程与故障排查见 [docs/user.md](docs/user.md).

## 功能特性

- **账号管理** — 支持扫码登录, Cookie 持久化与自动刷新
- **场次查询** — 查看项目详情, 票价, 场次, 实名制要求等信息
- **集群执行** — 支持自动管理本机 Worker, 并通过 gRPC + mTLS 接入远程 Worker
- **任务规划与调度** — 支持准点与回流阶段, 账号/Worker 资源分配, 多副本执行和故障转移
- **定时抢票** — 结合 Bilibili API 与 NTP 时钟校准执行定时任务
- **通知推送** — 支持 Bark, Gotify, PushPlus 三种通知渠道, 可同时启用多个
- **订单与支付保障** — 成功订单先持久化到雇主端, 再独立触发支付弹窗和消息推送
- **日志与状态** — 查看集群事件, Attempt 状态, Worker 健康状态和订单记录

## 成功订单与付款

- Worker 提交成功后, 雇主端会保存订单记录, 并在支付链接有效时打开支付二维码窗口.
- Bark, Gotify, PushPlus 等外部通知异步发送; 通知渠道超时或失败不会阻塞本地订单记录和支付弹窗.
- 同一逻辑订单启用多个执行副本时, 多个副本可能在停止信号生效前分别下单成功. 系统会逐笔记录并提示, 不会因为首个副本已经成功而忽略后续真实订单.
- 可随时进入 **集群 → 订单记录** 查看所有已知成功订单, 并在支付期限内重新打开付款窗口.

> 下单成功不等于付款成功. 准点抢票期间请同时留意支付弹窗, 系统通知和 "订单记录" 页面, 并在支付截止时间前逐笔确认需要支付的订单.
